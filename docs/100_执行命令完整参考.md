## 执行方式说明

MetaWeave 支持两种执行方式，效果完全一致：

### 方式一：`uv run`（推荐）

无需手动激活虚拟环境，`uv` 会自动管理。

```bash
uv run metaweave <子命令> [参数]
python -m metaweave.cli.main <子命令> [参数]
```

### 方式二：激活虚拟环境后直接运行

需要先激活对应环境，之后可直接调用 `metaweave` 或 `python -m metaweave.cli.main`。

```bash
# WSL / Linux 环境
source .venv-wsl/bin/activate

# Windows 环境（PowerShell）
.venv-win\Scripts\Activate.ps1

# Windows 环境（CMD）
.venv-win\Scripts\activate.bat
```

激活后可用以下两种等价方式执行：

```bash
# 直接调用入口命令（推荐）
metaweave <子命令> [参数]

# 或通过 python -m 调用
python -m metaweave.cli.main <子命令> [参数]
```

> 后续文档中各步骤命令块同时列出三种方式：**`uv run`、`metaweave`、`python -m metaweave.cli.main`**。

## 命令体系总览

```
metaweave [--debug] [--log-config PATH] 子命令
│
├── metadata    # 元数据生成主流程（单步骤 / standard 全流程）
├── pipeline    # 统一编排命令（两段式执行）
│   ├── generate   # 产物生成（9 步串行）
│   └── load       # 加载到目标库（3+1 步串行）
├── sql-rag     # SQL RAG 样例生成 / 校验 / 加载
├── load        # 将生成结果加载到目标数据库
└── dim_config  # 生成维度表配置文件
```

## 步骤依赖与数据流向

### 四条执行轨道

```
标准轨道（不依赖 LLM，metadata --step standard）：
  ddl ──► md ──► json ──► rel ──► cql

LLM 增强轨道（metadata 单步骤手动组合）：
  ddl ──► json_llm ──► rel_llm ──► cql
   │
   └──► md（可独立执行，不阻塞 json_llm）

Pipeline 轨道（推荐，pipeline generate + pipeline load）：
  ddl ──► md ──► generate-domains ──► json_llm ──► dim_config
   ──► rel_llm ──► cql ──► sql-rag-generate ──► sql-rag-validate
  （9 步串行，fail-fast，覆盖 LLM 增强 + SQL-RAG + 配置生成）

SQL-RAG 轨道（手动分步执行）：
  ddl ──► md ──► metadata --generate-domains ──► configs/db_domains.yaml
                  │
                  └──► sql-rag generate ──► output/sql/qs_{db}_pair.json
                               │
                               └──► sql-rag validate
                                      │
                                      ├──（enable_sql_repair=false）仅数据库校验 + 日志
                                      └──（enable_sql_repair=true）额外读取 output/rel/*.md 做修复
                                                   │
                                                   └──► load --type sql / sql-rag run-all
```

> `--step standard` 将 `md` 放在 `ddl` 之后、`json` 之前执行。

---

### 各步骤 / 命令输入 / 输出 / 依赖一览

| 步骤 | 读取（输入） | 写入（输出） | 需要 PostgreSQL | 需要 LLM |
|------|------------|------------|:--------------:|:--------:|
| `ddl` | PostgreSQL（直接查库） | `output/ddl/*.sql` | ✅ | ❌ |
| `md` | `output/ddl/*.sql` | `output/md/*.md` | ❌（完全 file-only） | ❌ |
| `metadata --generate-domains` | `output/md/*.md` + 可选 `--description` | `configs/db_domains.yaml` | ❌ | ✅ |
| `json` | `output/ddl/*.sql`（结构）+ PostgreSQL（COUNT/采样） | `output/json/*.json` | ✅ | ❌ |
| `json_llm` | 同 `json`（阶段 A）→ 再读 `output/json/*.json`（阶段 B） | `output/json/*.json`（原地覆写） | ✅ | ✅ |
| `rel` | `output/json/*.json` + PostgreSQL（采样验证） | `output/rel/*.json` + `.md` | ✅ | ❌ |
| `rel_llm` | `output/json/*.json` + PostgreSQL（采样验证） | `output/rel/*.json` + `.md` | ✅ | ✅ |
| `cql` / `cql_llm` | `output/json/*.json` + `output/rel/*.json` | `output/cql/*.cypher` + `.md` | ❌（完全 file-only） | ❌ |
| `sql-rag generate` | `configs/db_domains.yaml` + `output/md/*.md` | `output/sql/qs_{db}_pair.json` | ❌ | ✅ |
| `sql-rag validate` | `output/sql/qs_{db}_pair.json`；启用修复时额外读取 `output/rel/*.md` | 校验日志；启用修复时原地回写 `output/sql/qs_{db}_pair.json` | ✅ | 启用修复时需要 |
| `sql-rag run-all` | `configs/db_domains.yaml` + `output/md/*.md`；启用修复时额外读取 `output/rel/*.md` | 生成/校验后的 `output/sql/qs_{db}_pair.json` 并加载到 Milvus | ✅ | 生成阶段需要；修复开启时校验阶段也需要 |

> `json` 与 `json_llm` 写入**同一目录** `output/json/`；`rel` 与 `rel_llm` 写入同一目录 `output/rel/`。

---

### 步骤间强依赖关系

```
ddl
 ├──► md          （读 output/ddl/*.sql）
 │     └──► metadata --generate-domains   （读 output/md/*.md，写 configs/db_domains.yaml）
 │            └──► sql-rag generate       （读 configs/db_domains.yaml + output/md/*.md）
 │                   └──► sql-rag validate（读 output/sql/*.json；enable_sql_repair=true 时还读 output/rel/*.md）
 │                          └──► load --type sql / sql-rag run-all
 └──► json        （读 output/ddl/*.sql + 查库）
       └──► json_llm（在 json 基础上 LLM 写回，二选一）
             ↓
            rel / rel_llm   （读 output/json/*.json + 查库）
                   ↓
                  cql        （读 output/json/*.json + output/rel/*.json）
```

**关键约束**：

- `md`、`json`、`json_llm` 均依赖 `output/ddl/` 非空；若 DDL 目录为空则直接报错退出。
- `metadata --generate-domains` 依赖 `output/md/` 非空；推荐使用 `--description` 与 `--md-context-mode full` 提高生成质量。
- `sql-rag generate` 依赖 `configs/db_domains.yaml` 与 `output/md/`；`db_domains.yaml` 缺失时不会继续执行，因为 CLI 参数本身要求该文件存在。
- `sql-rag validate` 始终依赖 `output/sql/qs_{db}_pair.json`；只有 `enable_sql_repair=true` 时才额外依赖 `output/rel/*.md` 作为修复上下文。
- `rel`、`rel_llm`、`cql`、`cql_llm` 不依赖 DDL 文件，只依赖 JSON 文件。
- `cql`、`cql_llm` 完全 file-only，是整条流水线中唯一不需要数据库连接的 metadata 步骤（`md` 也不需要）。
- `json` 与 `json_llm` 互斥：两条轨道共享同一输出目录，后执行的会覆盖先执行的结果。

---

### `load` 命令的输入依赖

| 加载类型 | 读取来源 | 前置步骤 |
|---------|---------|---------|
| `load --type cql` | `output/cql/import_all.*.cypher` | `cql` 或 `cql_llm` |
| `load --type dim_value` | PostgreSQL 维度表（直接查库） | 无文件依赖 |
| `load --type table_schema` | `output/md/*.md` + `output/json/` | `md` 步骤 + `json` / `json_llm` 步骤 |
| `load --type sql` | `output/sql/qs_{db}_pair.json` | `sql-rag generate` / `validate` / `run-all` |

> `load --type table_schema` 默认读取 `output/json`，且按“MD 文件名同名 JSON（仅扩展名不同）”匹配输入。

## 一、`metadata` 命令

### 公共参数

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--config PATH` | `-c` | `configs/metadata_config.yaml` | 配置文件路径 |
| `--step STEP` | 无 | `standard` | 执行步骤：ddl/json/json_llm/cql/cql_llm/md/rel/rel_llm/standard |
| `--schemas TEXT` | `-s` | 全部 schema | 目标 schema，逗号分隔 |
| `--tables TEXT` | `-t` | 全部表 | 目标表名，逗号分隔 |
| `--max-workers N` | `-w` | `4` | 最大并发数 |
| `--incremental` | `-i` | false | ⚠️ 增量更新模式（**代码注释标注"暂未实现"，传入后不生效**） |
| `--clean` | 无 | false | 执行前清空该步骤的输出目录 |

---

### 业务域生成：`--generate-domains`

**需要**：LLM API + Markdown 摘要文件
**依赖**：`output/md/` 目录下有 `.md` 文件（通常需先执行 `--step md`）
**输出**：`configs/db_domains.yaml`（默认）

该命令基于现有的 Markdown 摘要，利用 LLM 自动推断数据库的业务主题（Domains）分类及数据库描述。

```bash
# 方式 A：uv run (推荐)
uv run metaweave metadata --generate-domains

# 方式 B：python -m 调用
python -m metaweave.cli.main metadata --generate-domains

# 推荐：配合数据库描述和完整 md 上下文执行
uv run metaweave metadata --generate-domains \
  --description "这是一个DVD影片租赁连锁店的数据" \
  --md-context-mode full
metaweave metadata --generate-domains \
  --description "这是一个DVD影片租赁连锁店的数据" \
  --md-context-mode full
python -m metaweave.cli.main metadata --generate-domains \
  --description "这是一个DVD影片租赁连锁店的数据" \
  --md-context-mode full
```

**可用参数列表**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--generate-domains` | (必填标志) | 启用业务域生成模式，完成后直接退出 |
| `--domains-config PATH` | `configs/db_domains.yaml` | 生成结果的保存路径 |
| `--description TEXT` | 无 | 可选：提供数据库背景补充说明，辅助 LLM 推断 |
| `--md-context-mode` | `name_comment` | 摘要模式：`name`(仅表名), `name_comment`(表名+首行), `full`(全文) |
| `--md-context-dir PATH` | `output/md` | Markdown 摘要文件的来源目录 |
| `--md-limit N` | `100` | 读取的 Markdown 文件数量上限 |

### Step 1：`ddl` — 提取 DDL 表结构

**需要**：PostgreSQL 数据库连接
**依赖**：无
**输出**：`output/ddl/{db}.{schema}.{table}.sql`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step ddl

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step ddl
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step ddl
```

---

### Step 2a：`json` — 生成数据画像（标准）

**需要**：PostgreSQL 数据库连接 + DDL 文件
**依赖**：`output/ddl/` 目录下有 `.sql` 文件（读取文件内容解析表结构，同时查库做 COUNT/采样）
**输出**：`output/json/{db}.{schema}.{table}.json`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step json

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step json
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step json
```

---

### Step 2b：`json_llm` — 生成数据画像（LLM 增强）

**需要**：PostgreSQL 数据库连接 + DDL 文件 + LLM API
**依赖**：`output/ddl/` 目录下有 `.sql` 文件（阶段A执行 `--step json`，会读取 DDL 内容并查库）
**输出**：`output/json/{db}.{schema}.{table}.json`（与 `json` **同一目录**，LLM 结果原地写回）

> **执行逻辑**：分两阶段
> 1. 阶段 A：完整执行 `--step json`（读取 DDL + 查库 COUNT/采样），生成数据画像
> 2. 阶段 B：LLM 对 JSON 文件进行增强（补全注释、推断分类、推断业务域），原地写回

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step json_llm

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step json_llm
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step json_llm
```

启用业务域功能（配合 `rel_llm` 使用）：

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step json_llm \
  --domain all \
  --domains-config configs/db_domains.yaml

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step json_llm \
  --domain all \
  --domains-config configs/db_domains.yaml
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step json_llm \
  --domain all \
  --domains-config configs/db_domains.yaml
```

**`json_llm` 特有参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--domain [all\|A,B]` | 无 | 业务域名称，仅在 `--generate-domains` 模式下生效；**普通 json_llm 主流程不使用此参数** |
| `--domains-config PATH` | `configs/db_domains.yaml` | 业务域定义文件路径 |
| `--generate-domains` | false | 仅生成域列表后退出（不执行 json 阶段 A/B，单独运行） |
| `--md-context` | false | 生成域列表时附加 md 目录摘要作为上下文 |
| `--md-context-dir PATH` | `output/md` | md 摘要来源目录 |
| `--md-context-mode` | `name_comment` | `name` / `name_comment` / `full` |
| `--md-limit N` | `100` | md 文件数量上限 |

单独生成业务域列表（不执行数据处理，直接退出）：

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml \
  --step json_llm \
  --generate-domains \
  --domains-config configs/db_domains.yaml

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml \
  --step json_llm \
  --generate-domains \
  --domains-config configs/db_domains.yaml
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml \
  --step json_llm \
  --generate-domains \
  --domains-config configs/db_domains.yaml
```

### Step 3a：`rel` — 发现表关系（标准算法）

**需要**：PostgreSQL 数据库连接 + JSON 文件
**依赖**：`output/json/` 目录下有 `.json` 文件
**输出**：`output/rel/{db}.relationships_global.json` 和 `.md`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step rel

metaweave metadata -c configs/metadata_config.yaml --step rel
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step rel
```

### Step 3b：`rel_llm` — 发现表关系（LLM 推断）

**需要**：PostgreSQL 数据库连接 + JSON 文件 + LLM API
**依赖**：`output/json/` 目录下有 `.json` 文件（与 `rel` 读取**同一目录**）
**输出**：`output/rel/{db}.relationships_global.json` 和 `.md`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step rel_llm

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step rel_llm
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step rel_llm
```

指定业务域范围：

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step rel_llm \
  --domain all \
  --cross-domain

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step rel_llm \
  --domain all \
  --cross-domain
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step rel_llm \
  --domain all \
  --cross-domain
```

**`rel_llm` 特有参数**：

| 参数 | 说明 |
|------|------|
| `--domain [all\|A,B]` | 只对指定域内的表进行 LLM 推断；要求 `db_domains.yaml` 中 `domains` 列表非空（可通过 `--generate-domains` 自动生成，也可手动维护该文件） |
| `--cross-domain` | 同时发现跨域关系 |

---

### Step 4：`cql` / `cql_llm` — 生成 Neo4j CQL 脚本

**需要**：仅文件（无需数据库连接）
**依赖**：`output/json/` + `output/rel/` 目录下有对应文件
**输出**：`output/cql/import_all.{db}.cypher` 和 `.md`

> ⚠️ **注意**：`cql` 与 `cql_llm` 代码完全相同，只有写入文件时的步骤名元数据不同，功能无区别。

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step cql
uv run metaweave metadata -c configs/metadata_config.yaml --step cql_llm  # 与上行等价

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step cql
metaweave metadata -c configs/metadata_config.yaml --step cql_llm
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step cql
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step cql_llm
```

---

### Step：`md` — 生成 Markdown 文档

**需要**：仅文件（**无需数据库连接**，完全 file-only）
**依赖**：`output/ddl/` 目录下有 `.sql` 文件（从 DDL 文件解析表结构和样本数据）
**输出**：`output/md/{db}.{schema}.{table}.md`

> **说明**：`md` 步骤在代码中明确跳过数据库连接，直接读取 DDL 文件生成 Markdown。

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step md

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step md
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step md
```

---

### 全流程执行：`standard`

> **代码核实的实际调度顺序**（固定，不可配置）：

| 序号 | `--step standard` |
|------|------------------|
| 1 | `ddl` |
| 2 | `md` |
| 3 | `json` |
| 4 | `rel` |
| 5 | `cql` |

> 任意子步骤失败则立即终止（fail-fast）。
>
> ⚠️ `--step standard` 仅执行标准（非 LLM）轨道。如需 LLM 增强全流程，请使用 `pipeline generate`（见下文第五节）。

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step standard

# 带清理（每个子步骤执行前清空其输出目录）
uv run metaweave metadata -c configs/metadata_config.yaml --step standard --clean

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step standard
metaweave metadata -c configs/metadata_config.yaml --step standard --clean
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step standard
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step standard --clean
```

## 二、`sql-rag` 命令

`sql-rag` 用于基于现有表结构文档与业务主题域配置，生成 Question-SQL 样例、校验 SQL，并将样例加载到 Milvus。

### 前置依赖

| 阶段 | 读取来源 | 前置条件 |
|------|---------|---------|
| `sql-rag generate` | `configs/db_domains.yaml` + `output/md/*.md` | 需先执行 `metadata --generate-domains`，且 `output/md/` 已生成 |
| `sql-rag validate` | `output/sql/qs_{db}_pair.json` | 需先执行 `sql-rag generate` |
| `sql-rag validate`（启用修复） | 额外读取 `output/rel/*.md` | 需先执行 `metadata --step rel` 或 `rel_llm` 生成关系文档 |
| `load --type sql` / `sql-rag run-all` | `output/sql/qs_{db}_pair.json` | 需先完成生成 / 校验 |

> **关于 `db_domains.yaml`**：
> - `sql-rag generate` 代码会直接读取 `configs/db_domains.yaml`，该文件应先通过 `metadata --generate-domains` 生成。
> - 若 `configs/db_domains.yaml` 缺失，命令不会继续执行，因为 `--domains-config` 参数要求文件必须存在。
> - 若文件存在但 `domains` 为空，命令可以启动，但无法按主题域生成有效样例，最终输出通常为空。

### 生成：`sql-rag generate`

**需要**：LLM + `configs/db_domains.yaml` + `output/md/`
**输出**：`output/sql/qs_{db}_pair.json`

```bash
# uv run
uv run metaweave sql-rag generate --clean

# 激活环境后
metaweave sql-rag generate --clean
python -m metaweave.cli.main sql-rag generate --clean
```

### 校验：`sql-rag validate`

**需要**：PostgreSQL + `output/sql/qs_{db}_pair.json`
**额外需要**（启用修复时）：LLM + `output/rel/`
**输出**：校验日志；启用修复时原地回写 `output/sql/qs_{db}_pair.json`

```bash
# uv run
uv run metaweave sql-rag validate --enable_sql_repair false

# 激活环境后
metaweave sql-rag validate --enable_sql_repair false
python -m metaweave.cli.main sql-rag validate --enable_sql_repair false
```

### 一键执行：`sql-rag run-all`

**执行顺序**：`generate -> validate -> load`
**需要**：`configs/db_domains.yaml` + `output/md/`；若启用修复，还需要 `output/rel/`

```bash
# uv run
uv run metaweave sql-rag run-all --clean

# 激活环境后
metaweave sql-rag run-all --clean
python -m metaweave.cli.main sql-rag run-all --clean
```

> `run-all --clean` 会同时：
> - 清理当前数据库旧的 `output/sql/qs_{db}_pair.json`
> - 在加载阶段清空目标 Milvus Collection

## 三、`load` 命令 — 加载到目标数据库

### 参数

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--type TYPE` | `-t` | 无（必填） | 加载类型（见下表） |
| `--config PATH` | `-c` | `configs/loader_config.yaml` | 加载器配置文件 |
| `--clean` | 无 | false | 加载前清空目标数据库（⚠️ 不可逆） |
| `--debug` | 无 | false | 开启调试模式 |

### 加载类型与实现状态

| `--type` | 目标数据库 | 数据来源 | 实现状态 |
|----------|-----------|---------|---------|
| `cql` | Neo4j（图数据库） | `output/cql/import_all.*.cypher` | ✅ 已实现 |
| `dim` / `dim_value` | Milvus（向量数据库） | PostgreSQL 维度表（直接查库） | ✅ 已实现 |
| `table_schema` | Milvus（向量数据库） | `output/md/` + `output/json/` | ✅ 已实现 |
| `sql` | Milvus（向量数据库） | `output/sql/qs_{db}_pair.json` | ✅ 已实现 |
| `content_md` | — | — | ❌ 未实现 |

> ⚠️ **`load --type cql` 通配符唯一性**：输入文件通过 `output/cql/import_all.*.cypher` 匹配，**必须有且只有一个匹配文件**，多个文件匹配时会直接失败。若切换过数据库或重复执行 `cql` 步骤，请先清理 `output/cql/` 中多余的 `.cypher` 文件。
>
> ⚠️ **向量 Loader 配置要求**：
> - `metadata_config_file` 建议在 `configs/loader_config.yaml` 顶层显式配置（未配置时代码默认回退 `configs/metadata_config.yaml`）。
> - `dim_loader.collection_name`、`table_schema_loader.collection_name` 与 `sql_loader.collection_name` 现在是必填项；缺失会直接报错并终止。

```bash
# uv run —— 将 CQL 脚本导入 Neo4j
uv run metaweave load --type cql -c configs/loader_config.yaml

# uv run —— 导入前先清空 Neo4j（危险操作，请谨慎）
uv run metaweave load --type cql --clean

# uv run —— 将维度表值向量化后写入 Milvus
uv run metaweave load --type dim_value -c configs/loader_config.yaml
uv run metaweave load --type dim_value -c configs/loader_config.yaml --clean

# uv run —— 将表结构（md + json）向量化后写入 Milvus
uv run metaweave load --type table_schema -c configs/loader_config.yaml
uv run metaweave load --type table_schema -c configs/loader_config.yaml --clean

# uv run —— 将 SQL Example（Question-SQL）向量化后写入 Milvus
uv run metaweave load --type sql -c configs/loader_config.yaml
uv run metaweave load --type sql -c configs/loader_config.yaml --clean

# 激活环境后（python -m 常用示例）

python -m metaweave.cli.main load --type cql --clean

python -m metaweave.cli.main load --type dim_value -c configs/loader_config.yaml --clean
```

## 五、`pipeline` 命令 — 两段式统一编排

`pipeline` 命令将多个步骤统一编排，提供"产物生成"（`generate`）和"数据加载"（`load`）两段式执行方式，是 **LLM 增强全流程的推荐入口**。

### `pipeline generate` — 产物生成（9 步串行）

**执行顺序**（固定，fail-fast）：

| 序号 | 步骤 | 说明 |
|------|------|------|
| 1 | `ddl` | 提取 DDL 表结构 |
| 2 | `md` | 生成 Markdown 文档 |
| 3 | `generate-domains` | LLM 推断业务域（写入 `db_domains.yaml`） |
| 4 | `json_llm` | 生成数据画像 + LLM 增强 |
| 5 | `dim_config` | 生成维度表配置（写入 `dim_tables.yaml`） |
| 6 | `rel_llm` | LLM 推断表关系 |
| 7 | `cql` | 生成 Neo4j CQL 脚本 |
| 8 | `sql-rag-generate` | 生成 Question-SQL 样例 |
| 9 | `sql-rag-validate` | 校验 SQL（启用修复时使用 LLM） |

**参数列表**：

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--config PATH` | `-c` | `configs/metadata_config.yaml` | metadata 配置文件路径 |
| `--sql-rag-config PATH` | 无 | `configs/sql_rag.yaml` | SQL RAG 配置文件路径 |
| `--loader-config PATH` | 无 | `configs/loader_config.yaml` | loader 配置文件路径（确定 dim_tables.yaml 目标路径） |
| `--domains-config PATH` | 无 | `configs/db_domains.yaml` | domain 配置文件路径 |
| `--description TEXT` | 无 | 无 | 可选：生成 db_domains.yaml 时的业务背景说明 |
| `--regenerate-configs` | 无 | false | 备份并重新生成配置型 YAML 文件（db_domains.yaml, dim_tables.yaml） |
| `--clean` | 无 | false | 清理 output/* 目录后重新生成 |
| `--debug` | 无 | false | 启用调试模式 |

> **配置文件处理策略**：
> - `db_domains.yaml` / `dim_tables.yaml` 默认 **skip-if-exists**：如果文件已存在，跳过生成步骤。
> - 使用 `--regenerate-configs` 会先备份再重新生成。

```bash
# uv run
uv run metaweave pipeline generate \
  -c configs/metadata_config.yaml \
  --description "这是一个DVD影片租赁连锁店的数据" \
  --clean

# 激活环境后
metaweave pipeline generate \
  -c configs/metadata_config.yaml \
  --description "这是一个DVD影片租赁连锁店的数据" \
  --clean

python -m metaweave.cli.main pipeline generate \
  -c configs/metadata_config.yaml \
  --description "这是一个DVD影片租赁连锁店的数据" \
  --clean

# 重新生成配置文件
uv run metaweave pipeline generate --regenerate-configs --clean
```

> **`pipeline generate` 参数常见问题**
>
> **Q1：`-c configs/metadata_config.yaml` 可以省略吗？**
> 可以省略。`--config` / `-c` 的默认值就是 `configs/metadata_config.yaml`（代码中 `default="configs/metadata_config.yaml"`），
> 省略后自动使用该路径，与显式传入效果完全一致。
>
> **Q2：`--clean` 会影响 `configs/` 目录下的 YAML 文件吗？**
> 不会。`--clean` 只清空 `output/` 的子目录（`ddl/`、`md/`、`json/`、`rel/`、`cql/`、`sql/`），
> `configs/` 目录下的任何文件均不在清理范围内，可以放心使用。
>
> **Q3：有参数控制是否覆盖 `configs/db_domains.yaml` 和 `dim_tables.yaml` 吗？**
> 有，就是 `--regenerate-configs`。
> - **不加该参数（默认）**：`db_domains.yaml` 和 `dim_tables.yaml` 若已存在，对应步骤直接**跳过**，不会覆盖，并打印警告提示。即便同时传入了 `--description`，该参数也不会生效——代码先判断文件是否存在，文件存在且未指定 `--regenerate-configs` 时直接 `return`，不会调用生成器。
> - **加 `--regenerate-configs`**：会先将旧文件备份为 `<原文件名>.bak_yyyyMMdd_HHmmss`，然后重新生成并覆盖。
> - `--clean` 与 `--regenerate-configs` 互不影响：前者针对 `output/`，后者针对 `configs/`，两个参数可以同时使用。

### `pipeline load` — 加载到目标库（3+1 步串行）

**执行顺序**（固定，fail-fast）：

| 序号 | 步骤 | 目标库 | 说明 |
|------|------|--------|------|
| 1 | `cql` | Neo4j | 导入 CQL 脚本 |
| 2 | `table_schema` | Milvus | 导入表结构向量 |
| 3 | `sql` | Milvus | 导入 Question-SQL 向量 |
| 4（可选） | `dim_value` | Milvus | 导入维度值向量（需 `--with-dim-values`） |

**参数列表**：

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--config PATH` | `-c` | `configs/metadata_config.yaml` | metadata 配置文件路径 |
| `--loader-config PATH` | 无 | `configs/loader_config.yaml` | loader 配置文件路径 |
| `--with-dim-values` | 无 | false | 是否加载 dim_value（需已确认 dim_tables.yaml） |
| `--clean` | 无 | false | 加载前清理目标库/集合 |
| `--debug` | 无 | false | 启用调试模式 |

```bash
# uv run
uv run metaweave pipeline load -c configs/metadata_config.yaml --clean

# 含 dim_value
uv run metaweave pipeline load --with-dim-values --clean

# 激活环境后
metaweave pipeline load -c configs/metadata_config.yaml --clean
metaweave pipeline load --with-dim-values --clean

python -m metaweave.cli.main pipeline load -c configs/metadata_config.yaml --clean
python -m metaweave.cli.main pipeline load --with-dim-values --clean
```

## 七、关键注意事项

| 注意点 | 说明 |
|--------|------|
| `json` 与 `json_llm` 输出同目录 | 都写入 `output/json/`，`json_llm` 是在 `json` 基础上原地覆盖 |
| `rel_llm` 读取 `output/json/` | 旧版 `json_llm_directory`（`output/json_llm/`）已废弃，两个步骤共用同一目录 |
| `md` 步骤完全 file-only | **不需要数据库连接**，从 DDL 文件读取，依赖 `output/ddl/` 中的 `.sql` 文件 |
| `pipeline generate` 覆盖了原 `all_llm` | 9 步串行，包含 generate-domains、dim_config、sql-rag 等扩展步骤 |
| `load --type cql` 通配符唯一性 | `output/cql/` 下只能有一个 `.cypher` 文件，多个时会直接失败，需手动清理 |
| `--incremental` 暂未实现 | 传入后不生效，代码注释明确标注"暂未实现" |
| `dim_config` 输出按数据库分组 | 生成 `databases.<database>.tables` 结构；`dim_value` 仅读取当前数据库节点 |
| `load` 不包含在 `standard` 中 | 需要单独执行；`pipeline load` 可统一编排加载步骤 |
| `sql-rag generate` 依赖 `db_domains.yaml` + `output/md/` | 需先执行 `metadata --generate-domains` 与 `metadata --step md` |

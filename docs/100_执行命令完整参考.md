## 执行方式说明

MetaWeave 支持两种执行方式，效果完全一致：

### 方式一：`uv run`（推荐）

无需手动激活虚拟环境，`uv` 会自动管理。

```bash
uv run metaweave <子命令> [参数]
python -m metaweave.cli.main <子命令> [参数]
```

### 方式二：激活虚拟环境后直接运行

需要先激活对应环境，之后可直接调用 `metaweave` 或 `python -m metaweave.cli.main`。

```bash
# WSL / Linux 环境
source .venv-wsl/bin/activate

# Windows 环境（PowerShell）
.venv-win\Scripts\Activate.ps1

# Windows 环境（CMD）
.venv-win\Scripts\activate.bat
```

激活后可用以下两种等价方式执行：

```bash
# 直接调用入口命令（推荐）
metaweave <子命令> [参数]

# 或通过 python -m 调用
python -m metaweave.cli.main <子命令> [参数]
```

> 后续文档中各步骤命令块同时列出三种方式：**`uv run`、`metaweave`、`python -m metaweave.cli.main`**。

```
└── dim_config  # 生成维度表配置文件
```

## 步骤依赖与数据流向

### 两条并行执行轨道

```
标准轨道（不依赖 LLM）：
  ddl ──► json ──► rel ──► cql
   │
   └──► md（可独立执行，不阻塞 json）

LLM 增强轨道：
  ddl ──► json_llm ──► rel_llm ──► cql
   │
   └──► md（同上）
```

> `--step all` 和 `--step all_llm` 均将 `md` 放在 `ddl` 之后、`json/json_llm` 之前执行。

---

### 各步骤输入 / 输出 / 依赖一览

| 步骤 | 读取（输入） | 写入（输出） | 需要 PostgreSQL | 需要 LLM |
|------|------------|------------|:--------------:|:--------:|
| `ddl` | PostgreSQL（直接查库） | `output/ddl/*.sql` | ✅ | ❌ |
| `md` | `output/ddl/*.sql` | `output/md/*.md` | ❌（完全 file-only） | ❌ |
| `json` | `output/ddl/*.sql`（结构）+ PostgreSQL（COUNT/采样） | `output/json/*.json` | ✅ | ❌ |
| `json_llm` | 同 `json`（阶段 A）→ 再读 `output/json/*.json`（阶段 B） | `output/json/*.json`（原地覆写） | ✅ | ✅ |
| `rel` | `output/json/*.json` + PostgreSQL（采样验证） | `output/rel/*.json` + `.md` | ✅ | ❌ |
| `rel_llm` | `output/json/*.json` + PostgreSQL（采样验证） | `output/rel/*.json` + `.md` | ✅ | ✅ |
| `cql` / `cql_llm` | `output/json/*.json` + `output/rel/*.json` | `output/cql/*.cypher` + `.md` | ❌（完全 file-only） | ❌ |

> `json` 与 `json_llm` 写入**同一目录** `output/json/`；`rel` 与 `rel_llm` 写入同一目录 `output/rel/`。

---

### 步骤间强依赖关系

```
ddl
 ├──► md          （读 output/ddl/*.sql）
 └──► json        （读 output/ddl/*.sql + 查库）
       └──► json_llm（在 json 基础上 LLM 写回，二选一）
             ↓
            rel / rel_llm   （读 output/json/*.json + 查库）
                   ↓
                  cql        （读 output/json/*.json + output/rel/*.json）
```

**关键约束**：

- `md`、`json`、`json_llm` 均依赖 `output/ddl/` 非空；若 DDL 目录为空则直接报错退出。
- `rel`、`rel_llm`、`cql`、`cql_llm` 不依赖 DDL 文件，只依赖 JSON 文件。
- `cql`、`cql_llm` 完全 file-only，是整条流水线中唯一不需要数据库连接的 metadata 步骤（`md` 也不需要）。
- `json` 与 `json_llm` 互斥：两条轨道共享同一输出目录，后执行的会覆盖先执行的结果。

---

### `load` 命令的输入依赖

| 加载类型 | 读取来源 | 前置步骤 |
|---------|---------|---------|
| `load --type cql` | `output/cql/import_all.*.cypher` | `cql` 或 `cql_llm` |
| `load --type dim_value` | PostgreSQL 维度表（直接查库） | 无文件依赖 |
| `load --type table_schema` | `output/md/*.md` + `output/json/` | `md` 步骤 + `json` / `json_llm` 步骤 |

> `load --type table_schema` 默认读取 `output/json`，且按“MD 文件名同名 JSON（仅扩展名不同）”匹配输入。

## 一、`metadata` 命令

### 公共参数

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--config PATH` | `-c` | `configs/metadata_config.yaml` | 配置文件路径 |
| `--step STEP` | 无 | `all` | 执行步骤（见下方各步骤说明） |
| `--schemas TEXT` | `-s` | 全部 schema | 目标 schema，逗号分隔 |
| `--tables TEXT` | `-t` | 全部表 | 目标表名，逗号分隔 |
| `--max-workers N` | `-w` | `4` | 最大并发数 |
| `--incremental` | `-i` | false | ⚠️ 增量更新模式（**代码注释标注"暂未实现"，传入后不生效**） |
| `--clean` | 无 | false | 执行前清空该步骤的输出目录 |

---

### 业务域生成：`--generate-domains`

**需要**：LLM API + Markdown 摘要文件
**依赖**：`output/md/` 目录下有 `.md` 文件（通常需先执行 `--step md`）
**输出**：`configs/db_domains.yaml`（默认）

该命令基于现有的 Markdown 摘要，利用 LLM 自动推断数据库的业务主题（Domains）分类及数据库描述。

```bash
# 方式 A：uv run (推荐)
uv run metaweave metadata --generate-domains

# 方式 B：python -m 调用
python -m metaweave.cli.main metadata --generate-domains

# 配合补充描述执行
uv run metaweave metadata --generate-domains --description "这是一个电商系统"
python -m metaweave.cli.main metadata --generate-domains --description "这是一个电商系统"
```

**可用参数列表**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--generate-domains` | (必填标志) | 启用业务域生成模式，完成后直接退出 |
| `--domains-config PATH` | `configs/db_domains.yaml` | 生成结果的保存路径 |
| `--description TEXT` | 无 | 可选：提供数据库背景补充说明，辅助 LLM 推断 |
| `--md-context-mode` | `name_comment` | 摘要模式：`name`(仅表名), `name_comment`(表名+首行), `full`(全文) |
| `--md-context-dir PATH` | `output/md` | Markdown 摘要文件的来源目录 |
| `--md-limit N` | `100` | 读取的 Markdown 文件数量上限 |

---

### Step 1：`ddl` — 提取 DDL 表结构

**需要**：PostgreSQL 数据库连接
**依赖**：无
**输出**：`output/ddl/{db}.{schema}.{table}.sql`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step ddl

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step ddl
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step ddl
```

---

### Step 2a：`json` — 生成数据画像（标准）

**需要**：PostgreSQL 数据库连接 + DDL 文件
**依赖**：`output/ddl/` 目录下有 `.sql` 文件（读取文件内容解析表结构，同时查库做 COUNT/采样）
**输出**：`output/json/{db}.{schema}.{table}.json`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step json

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step json
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step json
```

---

### Step 2b：`json_llm` — 生成数据画像（LLM 增强）

**需要**：PostgreSQL 数据库连接 + DDL 文件 + LLM API
**依赖**：`output/ddl/` 目录下有 `.sql` 文件（阶段A执行 `--step json`，会读取 DDL 内容并查库）
**输出**：`output/json/{db}.{schema}.{table}.json`（与 `json` **同一目录**，LLM 结果原地写回）

> **执行逻辑**：分两阶段
> 1. 阶段 A：完整执行 `--step json`（读取 DDL + 查库 COUNT/采样），生成数据画像
> 2. 阶段 B：LLM 对 JSON 文件进行增强（补全注释、推断分类、推断业务域），原地写回

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step json_llm

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step json_llm
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step json_llm
```

启用业务域功能（配合 `rel_llm` 使用）：

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step json_llm \
  --domain all \
  --domains-config configs/db_domains.yaml

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step json_llm \
  --domain all \
  --domains-config configs/db_domains.yaml
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step json_llm \
  --domain all \
  --domains-config configs/db_domains.yaml
```

**`json_llm` 特有参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--domain [all\|A,B]` | 无 | 业务域名称，仅在 `--generate-domains` 模式下生效；**普通 json_llm 主流程不使用此参数** |
| `--domains-config PATH` | `configs/db_domains.yaml` | 业务域定义文件路径 |
| `--generate-domains` | false | 仅生成域列表后退出（不执行 json 阶段 A/B，单独运行） |
| `--md-context` | false | 生成域列表时附加 md 目录摘要作为上下文 |
| `--md-context-dir PATH` | `output/md` | md 摘要来源目录 |
| `--md-context-mode` | `name_comment` | `name` / `name_comment` / `full` |
| `--md-limit N` | `100` | md 文件数量上限 |

单独生成业务域列表（不执行数据处理，直接退出）：

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml \
  --step json_llm \
  --generate-domains \
  --domains-config configs/db_domains.yaml

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml \
  --step json_llm \
  --generate-domains \
  --domains-config configs/db_domains.yaml
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml \
  --step json_llm \
  --generate-domains \
  --domains-config configs/db_domains.yaml
```

---

### Step 3a：`rel` — 发现表关系（标准算法）

**需要**：PostgreSQL 数据库连接 + JSON 文件
**依赖**：`output/json/` 目录下有 `.json` 文件
**输出**：`output/rel/{db}.relationships_global.json` 和 `.md`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step rel

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step rel
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step rel
```

---

### Step 3b：`rel_llm` — 发现表关系（LLM 推断）

**需要**：PostgreSQL 数据库连接 + JSON 文件 + LLM API
**依赖**：`output/json/` 目录下有 `.json` 文件（与 `rel` 读取**同一目录**）
**输出**：`output/rel/{db}.relationships_global.json` 和 `.md`

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step rel_llm

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step rel_llm
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step rel_llm
```

指定业务域范围：

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step rel_llm \
  --domain all \
  --cross-domain

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step rel_llm \
  --domain all \
  --cross-domain
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step rel_llm \
  --domain all \
  --cross-domain
```

**`rel_llm` 特有参数**：

| 参数 | 说明 |
|------|------|
| `--domain [all\|A,B]` | 只对指定域内的表进行 LLM 推断；要求 `db_domains.yaml` 中 `domains` 列表非空（可通过 `--generate-domains` 自动生成，也可手动维护该文件） |
| `--cross-domain` | 同时发现跨域关系 |

---

### Step 4：`cql` / `cql_llm` — 生成 Neo4j CQL 脚本

**需要**：仅文件（无需数据库连接）
**依赖**：`output/json/` + `output/rel/` 目录下有对应文件
**输出**：`output/cql/import_all.{db}.cypher` 和 `.md`

> ⚠️ **注意**：`cql` 与 `cql_llm` 代码完全相同，只有写入文件时的步骤名元数据不同，功能无区别。

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step cql
uv run metaweave metadata -c configs/metadata_config.yaml --step cql_llm  # 与上行等价

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step cql
metaweave metadata -c configs/metadata_config.yaml --step cql_llm
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step cql
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step cql_llm
```

---

### Step：`md` — 生成 Markdown 文档

**需要**：仅文件（**无需数据库连接**，完全 file-only）
**依赖**：`output/ddl/` 目录下有 `.sql` 文件（从 DDL 文件解析表结构和样本数据）
**输出**：`output/md/{db}.{schema}.{table}.md`

> **说明**：`md` 步骤在代码中明确跳过数据库连接，直接读取 DDL 文件生成 Markdown。

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step md

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step md
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step md
```

---

### 全流程执行：`all` / `all_llm`

> **代码核实的实际调度顺序**（固定，不可配置）：

| 序号 | `--step all` | `--step all_llm` |
|------|-------------|-----------------|
| 1 | `ddl` | `ddl` |
| 2 | `md` | `md` |
| 3 | `json` | `json_llm` |
| 4 | `rel` | `rel_llm` |
| 5 | `cql` | `cql`（不是 `cql_llm`） |

> 任意子步骤失败则立即终止（fail-fast）。

```bash
# uv run
uv run metaweave metadata -c configs/metadata_config.yaml --step all
uv run metaweave metadata -c configs/metadata_config.yaml --step all_llm

# 带清理（每个子步骤执行前清空其输出目录）
uv run metaweave metadata -c configs/metadata_config.yaml --step all --clean
uv run metaweave metadata -c configs/metadata_config.yaml --step all_llm --clean

# 激活环境后
metaweave metadata -c configs/metadata_config.yaml --step all
metaweave metadata -c configs/metadata_config.yaml --step all_llm
metaweave metadata -c configs/metadata_config.yaml --step all --clean
metaweave metadata -c configs/metadata_config.yaml --step all_llm --clean
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step all
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step all_llm
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step all --clean
python -m metaweave.cli.main metadata -c configs/metadata_config.yaml --step all_llm --clean
```

## 二、`load` 命令 — 加载到目标数据库

### 参数

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--type TYPE` | `-t` | 无（必填） | 加载类型（见下表） |
| `--config PATH` | `-c` | `configs/loader_config.yaml` | 加载器配置文件 |
| `--clean` | 无 | false | 加载前清空目标数据库（⚠️ 不可逆） |
| `--debug` | 无 | false | 开启调试模式 |

### 加载类型与实现状态

| `--type` | 目标数据库 | 数据来源 | 实现状态 |
|----------|-----------|---------|---------|
| `cql` | Neo4j（图数据库） | `output/cql/import_all.*.cypher` | ✅ 已实现 |
| `dim` / `dim_value` | Milvus（向量数据库） | PostgreSQL 维度表（直接查库） | ✅ 已实现 |
| `table_schema` | Milvus（向量数据库） | `output/md/` + `output/json/` | ✅ 已实现 |
| `content_md` | — | — | ❌ 未实现 |
| `sql` | — | — | ❌ 未实现 |

> ⚠️ **`load --type cql` 通配符唯一性**：输入文件通过 `output/cql/import_all.*.cypher` 匹配，**必须有且只有一个匹配文件**，多个文件匹配时会直接失败。若切换过数据库或重复执行 `cql` 步骤，请先清理 `output/cql/` 中多余的 `.cypher` 文件。
>
> ⚠️ **向量 Loader 配置要求**：
> - `metadata_config_file` 建议在 `configs/loader_config.yaml` 顶层显式配置（未配置时代码默认回退 `configs/metadata_config.yaml`）。
> - `dim_loader.collection_name` 与 `table_schema_loader.collection_name` 现在是必填项；缺失会直接报错并终止。

```bash
# uv run —— 将 CQL 脚本导入 Neo4j
uv run metaweave load --type cql -c configs/loader_config.yaml

# uv run —— 导入前先清空 Neo4j（危险操作，请谨慎）
uv run metaweave load --type cql --clean

# uv run —— 将维度表值向量化后写入 Milvus
uv run metaweave load --type dim_value -c configs/loader_config.yaml
uv run metaweave load --type dim_value -c configs/loader_config.yaml --clean

# uv run —— 将表结构（md + json）向量化后写入 Milvus
uv run metaweave load --type table_schema -c configs/loader_config.yaml
uv run metaweave load --type table_schema -c configs/loader_config.yaml --clean

# 激活环境后（python -m 常用示例）
metaweave load --type cql -c configs/loader_config.yaml
metaweave load --type cql --clean
metaweave load --type dim_value -c configs/loader_config.yaml
metaweave load --type dim_value -c configs/loader_config.yaml --clean
metaweave load --type table_schema -c configs/loader_config.yaml
metaweave load --type table_schema -c configs/loader_config.yaml --clean
python -m metaweave.cli.main load --type cql -c configs/loader_config.yaml
python -m metaweave.cli.main load --type cql --clean
python -m metaweave.cli.main load --type dim_value -c configs/loader_config.yaml
python -m metaweave.cli.main load --type dim_value -c configs/loader_config.yaml --clean
python -m metaweave.cli.main load --type table_schema -c configs/loader_config.yaml
python -m metaweave.cli.main load --type table_schema -c configs/loader_config.yaml --clean
```

## 三、`dim_config` 命令 — 生成维度表配置

从 `output/json` 扫描 `table_category='dim'` 的表，生成 `dim_tables.yaml`。

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--generate` | 无（必填标志） | 执行生成操作 |
| `--config PATH` | `configs/metadata_config.yaml` | 元数据配置文件 |
| `--output PATH` | `configs/dim_tables.yaml` | 输出文件路径 |

> ⚠️ **前置条件（重要）**：`dim_config` 读取 `metadata_config.yaml` 中的 `output.json_directory`，
> 并扫描该目录下的 `*.json` 文件来识别 `table_profile.table_category='dim'` 的维表。
> 生成文件结构为 `databases.<database>.tables.<schema.table>`；`load --type dim_value` 仅加载当前数据库节点。

```bash
# uv run
uv run metaweave dim_config --generate \
  -c configs/metadata_config.yaml \
  -o configs/dim_tables.yaml

# 激活环境后
metaweave dim_config --generate \
  -c configs/metadata_config.yaml \
  -o configs/dim_tables.yaml
python -m metaweave.cli.main dim_config --generate \
  -c configs/metadata_config.yaml \
  -o configs/dim_tables.yaml
# 等价单行写法
python -m metaweave.cli.main dim_config --generate -c configs/metadata_config.yaml -o configs/dim_tables.yaml
```

> 生成后需**手动填写** `embedding_col` 字段（指定要向量化的列名）。

## 四、推荐执行流程

### 标准流程（不依赖 LLM）

## 五、关键注意事项

| 注意点 | 说明 |
|--------|------|
| `json` 与 `json_llm` 输出同目录 | 都写入 `output/json/`，`json_llm` 是在 `json` 基础上原地覆盖 |
| `rel_llm` 读取 `output/json/` | 旧版 `json_llm_directory`（`output/json_llm/`）已废弃，两个步骤共用同一目录 |
| `md` 步骤完全 file-only | **不需要数据库连接**，从 DDL 文件读取，依赖 `output/ddl/` 中的 `.sql` 文件 |
| `all_llm` 最后一步是 `cql` | 不是 `cql_llm`（两者代码等价） |
| `load --type cql` 通配符唯一性 | `output/cql/` 下只能有一个 `.cypher` 文件，多个时会直接失败，需手动清理 |
| `--incremental` 暂未实现 | 传入后不生效，代码注释明确标注"暂未实现" |
| `dim_config` 输出按数据库分组 | 生成 `databases.<database>.tables` 结构；`dim_value` 仅读取当前数据库节点 |
| `load` 不包含在 `all` 中 | 需要单独执行 |
| 调试模式 | 子命令前加 `--debug`，例如：`uv run metaweave --debug metadata ...` |
| WSL 使用 `.venv-wsl` | Windows 本地使用 `.venv-win`，两套环境不可混用 |

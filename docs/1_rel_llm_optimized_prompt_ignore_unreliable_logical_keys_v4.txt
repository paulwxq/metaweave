## 背景知识

### 关于提供的 JSON 数据
这些 JSON 来自完整的表画像分析，包含：
- **physical_constraints**：数据库物理约束（主键、外键、唯一约束），这是最可靠的信息
- **semantic_analysis**：列的语义角色分析（identifier/metric/datetime/audit/enum/description/attribute），附带置信度
- **table_category**：表类型分类（fact/dim/bridge/unknown），附带置信度
- **statistics**：基于采样数据的统计信息（uniqueness、null_rate、value_distribution 等）
- **sample_records**：样例数据记录
- **logical_keys**：基于采样推断的逻辑主键候选（可能不准确，仅供参考）

### 典型的关联模式
- **Fact → Dim**：事实表引用维度表（最常见），例如订单表的 customer_id 引用客户维度表的主键
- **Bridge → Dim/Fact**：桥接表连接其他表，通常包含两个或多个外键
- **Dim → Dim**：维度层级关系，例如城市维度表的 province_id 引用省份维度表
- **不太常见**：Dim → Fact（维度表引用事实表，罕见）、Metric ↔ Metric（度量字段互相引用，通常不合理）

### 判断提示（非强制要求）
你可以综合考虑以下线索：
1. **物理约束**：如果 physical_constraints.foreign_keys 已明确声明，直接输出即可
2. **字段命名**：相同或相似的字段名（如 customer_id ↔ customer_id，或 store_id ↔ shop_id）
3. **语义角色**：semantic_role 为 "identifier" 的字段更可能用于关联
4. **数据类型**：兼容的数据类型（integer ↔ integer，varchar ↔ varchar）
5. **样例数据值域**：一个表的字段值是否是另一个表字段值的子集（暗示外键关系）
6. **表类型关系**：fact 表通常引用 dim 表，bridge 表通常连接两个其他表
7. **业务语义**：字段注释（comment）中的业务含义是否相关

### 注意事项
- logical_keys 是基于采样推断的，可能不准确，不要完全依赖它
- statistics 中的 uniqueness 和 is_unique 也是基于采样的，可能受样本量影响
- sample_records 可以帮助验证值域匹配，但样本可能不完整
- 不要强行输出关系，如果不确定就返回空数组


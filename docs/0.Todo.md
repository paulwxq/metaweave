--- 关于缓存的问题

• 1. 缓存在哪个阶段生成/更新？

- 只有走 CommentGenerator 的路径才会读写缓存：也就是 MetadataGenerator 的 数据库路径（metaweave/core/metadata/generator.py 里 _process_table_from_db() 会在 self.comment_enabled 为真时调用 self.comment_generator.enrich_metadata_with_comments(...)）。
- 因此通常是 --step ddl / md / all（以及当前实现里 --step cql 也会走 DB 路径）会生成/更新缓存；--step json 走 DDL 路径，不调用 CommentGenerator，不会更新缓存。
- 当前 --step json_llm（旧 LLMJsonGenerator）不使用 CacheService，不会写 cache/comment_cache.json。
  - 整表 columns 缓存不存在：会一次性调用 LLM 为“本次检测到缺失注释的字段集合”生成注释，然后把整表字典写入缓存（不是单字段）。
  - 整表 columns 缓存存在但缺了某个字段：当前代码会直接返回整表缓存字典，不会为“缺的那一个字段”再补打一发 LLM，所以那个字段会继续缺注释（除非你禁用缓存/清缓存/强制重生成；但 enrich_metadata_with_comments 目前不暴露 force_regenerate）。

  1. 缓存清理机制是什么？什么时候会清除？

- 代码里没有自动清理/失效机制，也不会因为 schema/采样变化自动刷新。
- CacheService 支持 TTL 和 cleanup_expired()（metaweave/services/cache_service.py），但当前 CommentGenerator 写缓存时没传 TTL，所以默认都是“永久缓存”；而且项目里也没有调用 cleanup_expired()、clear()、delete()。
- 实际清理只能靠你手动做：删除 cache/comment_cache.json，或把 comment_generation.cache_enabled: false 来禁用缓存读写。

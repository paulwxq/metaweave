    """将 dict 格式的关系转换为 Relation 对象

    注意：dict 格式遵循 rel JSON 约定（from=主键表, to=外键表），

## 关键设计决策

### 1. 语义转换处理

**问题：** dict 格式和 Relation 对象的语义相反
- dict: `from=主键表, to=外键表`
- Relation: `source=外键表, target=主键表`

**解决方案：**
- `_dict_to_relation()`: dict → Relation 时交换 from/to
- `_relation_to_dict()`: Relation → dict 时交换 source/target
- 同时翻转 cardinality（N:1 ↔ 1:N）

### 2. 字段名映射

| dict 格式 | Relation 对象 |
|----------|--------------|
| `metrics` | `score_details` |
| `discovery_method` | `inference_method` |
| `from_table` | `target_*` (主键表) |
| `to_table` | `source_*` (外键表) |

### 3. 向后兼容

- RelationshipWriter 的 `generated_by` 默认值为 `"rel"`，确保 `--step rel` 不受影响
- `extra_statistics` 为可选参数，默认为 `None`
- CQL 生成器只读取 relationships 数组中的必需字段，新字段不影响

## 关键 Bug 修复

### Bug 1: `_filter_existing_fks()` 单向检查失效 🔴

**严重性：** 高优先级
**影响：** 物理外键可能进入 LLM 评分阶段，造成资源浪费和潜在的重复关系

**问题描述：**
代码错误地假设 `relationship_id` 是双向一致的，但实际上它是有方向的（source->target 格式）。当 LLM 返回与物理外键相反的方向时，过滤失效。

**修复：** `metaweave/core/relationships/llm_relationship_discovery.py:898-960`
```python
# 计算正向和反向两个 relationship_id

    source=from_table, target=to_table, ...
)
reverse_rel_id = MetadataRepository.compute_relationship_id(
    source=to_table, target=from_table, ...  # 交换方向
)

# 检查任一方向匹配

    # 跳过这个候选（它是物理外键）
```

### Bug 2: `_dict_to_relation()` 使用错误的 relationship_id 方向 🔴

**严重性：** 高优先级
**影响：** 阶段7去重完全失效，所有物理外键会被重复输出（一次作为 foreign_key，一次作为 llm_assisted）

**问题描述：**
方法直接复制 dict 的 `relationship_id`（使用 PK->FK 方向），但 Relation 对象应使用 FK->PK 方向。导致物理外键和 LLM 推断关系的 ID 永远不匹配。

**修复：** `metaweave/core/relationships/llm_relationship_discovery.py:616-663`

### Enhancement 8: LLM 响应解析增强（多模式提取 + 状态机）✅

**日期**: 2025-12-26
**优先级**: 高优先级（真实 bug 风险）
**位置**: `metaweave/core/relationships/llm_relationship_discovery.py:733-907`

**问题**:
原解析器使用简单括号计数法，**不处理字符串内的花括号**，可能导致：
- 截断错误：字符串内的 `}` 被误认为 JSON 结束
- 解析失败：提取到不完整的 JSON
- 示例 Bug: `{"reason": "Pattern: {FK} -> {PK}"}` 会在第一个 `}` 处截断

**原有缺陷代码**:
```python
# 简单计数花括号（不处理字符串内的 {/}）
for i in range(start_idx, len(cleaned_response)):
    if cleaned_response[i] == '{':  # ❌ 字符串内的 { 也会计数
        brace_count += 1
    elif cleaned_response[i] == '}':  # ❌ 字符串内的 } 也会计数
        brace_count -= 1
        if brace_count == 0:
            break  # 可能过早停止
```

**解决方案 - 多模式提取（4层降级）**:

```python
def _parse_llm_response(self, response: str) -> List[Dict]:
    """解析 LLM 返回（增强版，多模式提取）

    提取优先级：
    1. ```json ... ``` 代码块（最可靠，处理 95%+ 场景）
    2. ``` ... ``` 无语言标签代码块
    3. 首个完整 JSON 对象（状态机，正确处理字符串）
    4. 降级：简单 brace_count（向后兼容）
    """

    # 方法 1: ```json ... ```
    json_block_pattern = r'```json\s*\n(.*?)\n```'
    match = re.search(json_block_pattern, response, re.DOTALL | re.IGNORECASE)
    if match:
        data = json.loads(match.group(1).strip())
        if self._validate_response_structure(data):
            return data.get("relationships", [])

    # 方法 2: ``` ... ```
    # ... 类似 ...

    # 方法 3: 状态机提取
    json_obj = self._extract_first_json_object(response)
    if json_obj:
        data = json.loads(json_obj)
        if self._validate_response_structure(data):
            return data.get("relationships", [])

    # 方法 4: Fallback（原逻辑）
    # ...
```

**核心改进 - 状态机提取**:
```python
def _extract_first_json_object(self, text: str) -> Optional[str]:
    """使用状态机提取首个完整 JSON 对象

    状态机逻辑：
    - 跟踪是否在字符串内（in_string）
    - 处理转义字符（escape_next）
    - 只在非字符串内计数花括号
    """
    in_string = False
    escape_next = False
    brace_count = 0

    for i, char in enumerate(text):
        if escape_next:
            escape_next = False
            continue

        if char == '\\' and in_string:
            escape_next = True
            continue

        if char == '"':
            in_string = not in_string  # ✅ 跟踪字符串边界
            continue

        # ✅ 只在非字符串内计数
        if not in_string:
            if char == '{':
                brace_count += 1
            elif char == '}':
                brace_count -= 1
                if brace_count == 0:
                    return text[start_idx:i+1]
    return None
```

### Fix 9: datetime.utcnow() DeprecationWarning 修复 ✅

**日期**: 2025-12-26
**优先级**: 中优先级（不影响功能，避免未来兼容性问题）
**位置**: 6个文件，6处修改

**问题**:
`datetime.utcnow()` 产生 DeprecationWarning，即将在未来 Python 版本中移除。

**Warning 信息**:
```
DeprecationWarning: datetime.datetime.utcnow() is deprecated and scheduled
for removal in a future version. Use timezone-aware objects to represent
datetimes in UTC: datetime.datetime.now(datetime.UTC).
```

**问题本质**:
- `utcnow()` 返回 naive datetime（无时区信息）
- 手工拼接 "Z" 只是表面规范
- 容易与本地时区的 naive datetime 混用

**修复前**:
```python
from datetime import datetime

timestamp = datetime.utcnow().isoformat() + "Z"  # ❌ 产生 warning
```

**修复后**:
```python
from datetime import datetime, timezone

timestamp = datetime.now(timezone.utc).isoformat().replace("+00:00", "Z")  # ✅ 无 warning
```

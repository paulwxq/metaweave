1. 命令概述

metaweave metadata --step rel_llm 命令使用 LLM（大语言模型）辅助发现数据库表之间的隐式关系，结合传统规则和 AI 语义理解能力。

核心特点

  • 🤖 LLM 语义推断：利用 LLM 理解表结构和字段语义，发现隐式关系
  • 📊 数据库验证：通过数据采样验证 LLM 推断的关系
  • 🔗 外键直通：保留物理外键关系，与 LLM 推断关系合并
  • 🎯 Domain 过滤：支持按业务域过滤表对，减少 LLM 调用次数
  • ⚡ 异步批处理：支持异步并发调用 LLM，提高效率

与 rel 命令的区别

特性 rel（规则驱动） rel_llm（LLM 辅助）
候选生成 基于物理约束和逻辑主键 LLM 语义推断
数据库访问 候选生成 + 评分 仅评分阶段
发现能力 依赖约束和命名规则 理解语义和业务逻辑
性能 快速（无 LLM 调用） 较慢（需调用 LLM）
成本 无额外成本 LLM API 调用成本
💡 适用场景：
  • 表之间缺少物理外键约束
  • 列名不遵循命名规范
  • 需要发现复杂的业务关系
  • 希望利用 AI 提高关系发现的召回率

get_table_pairs()
📁 metaweave/core/relationships/llm_relationship_discovery.py:168
根据 domain 和 cross_domain 参数生成需要 LLM 分析的表对组合。
表对生成策略：
  1. 无过滤（domain=None, cross_domain=False）：
    • 生成所有表的两两组合（C(n,2)）
  2. 域内关系（domain="A,B", cross_domain=False）：
    • 只生成同一 domain 内的表对
    • 例如：sales 域内的表两两组合
  3. 域内+跨域（domain="A,B", cross_domain=True）：
    • 生成域内表对 + 跨域表对
    • 例如：sales 域内 + sales↔inventory 跨域
  4. 仅跨域（domain=None, cross_domain=True）：
    • 只生成不同 domain 之间的表对
⚠️ 性能优化： 使用 domain 过滤可以显著减少 LLM 调用次数。例如，100 张表的全组合是 4950 对，但按 5 个 domain 分组后可能只需要几百对。

7.2.1 维度1: inclusion_rate（数据包含率）

RelationshipScorer._sample_and_calculate_inclusion()
📁 metaweave/core/relationships/scorer.py:246
从数据库采样并计算包含率和 Jaccard 指数。
算法步骤：
  1. 采样源表：SELECT source_columns FROM source_table LIMIT 1000
  2. 采样目标表：SELECT target_columns FROM target_table LIMIT 1000
  3. 提取值集合：
    • 单列：直接提取列值（排除 NULL）
    • 复合列：组合为元组（如 (order_id, product_id)）
  4. 计算交集：intersection = source_values ∩ target_values
  5. 计算包含率：inclusion_rate = |intersection| / |source_values|
计算示例：
源表 employee 采样 1000 行，dept_id 值集合：
source_values = {1, 2, 3, 4, 5}  # 5 个不同值

目标表 department 采样 1000 行，dept_id 值集合：
target_values = {1, 2, 3, 6, 7}  # 5 个不同值

交集：
intersection = {1, 2, 3}  # 3 个共同值

包含率：
inclusion_rate = 3 / 5 = 0.60 (60%)
⚠️ NULL 值处理： 包含 NULL 的行会被排除，只计算有效值。如果 NULL 率超过 50%，会记录警告。

7.2.2 维度2: jaccard_index（Jaccard 相似度）

RelationshipScorer._sample_and_calculate_inclusion()
📁 metaweave/core/relationships/scorer.py:305
计算 Jaccard 相似度，衡量两个集合的重叠程度。
算法公式：
  1. 计算并集：union = source_values ∪ target_values
  2. 计算 Jaccard：jaccard_index = |intersection| / |union|
计算示例：
使用上面相同的数据：
source_values = {1, 2, 3, 4, 5}
target_values = {1, 2, 3, 6, 7}
intersection = {1, 2, 3}

并集：
union = {1, 2, 3, 4, 5, 6, 7}  # 7 个不同值

Jaccard 相似度：
jaccard_index = 3 / 7 = 0.43 (43%)
💡 与 inclusion_rate 的区别：
  • inclusion_rate：关注源表值在目标表中的覆盖率（外键视角）
  • jaccard_index：关注两个集合的整体相似度（对称性）

7.2.3 维度3: name_similarity（列名相似度）

RelationshipScorer._calculate_name_similarity()
📁 metaweave/core/relationships/scorer.py:511
计算列名相似度，使用 Levenshtein 算法（SequenceMatcher）。
算法步骤：
  1. 检查列数：源列和目标列数量必须相等，否则返回 0.0
  2. 逐对计算：对每对列名计算相似度
    • 完全相同：similarity = 1.0
    • 不同：使用 SequenceMatcher.ratio() 计算
  3. 平均值：avg_similarity = sum(similarities) / len(columns)
计算示例：
单列示例：
source_column = "dept_id"
target_column = "department_id"

SequenceMatcher.ratio("dept_id", "department_id") = 0.73

复合列示例：
source_columns = ["order_id", "product_id"]
target_columns = ["order_id", "prod_id"]

逐对计算：
- "order_id" vs "order_id": 1.0 (完全相同)
- "product_id" vs "prod_id": 0.73

平均值：
name_similarity = (1.0 + 0.73) / 2 = 0.865
💡 SequenceMatcher 算法： 基于 Levenshtein 距离，计算两个字符串的相似度（0-1），考虑字符的插入、删除、替换操作。

7.2.4 维度4: type_compatibility（类型兼容性）

RelationshipScorer._calculate_type_compatibility()
📁 metaweave/core/relationships/scorer.py:541
计算数据类型兼容性，评估 JOIN 操作的性能。
算法步骤：
  1. 检查列数：源列和目标列数量必须相等，否则返回 0.0
  2. 逐对计算：对每对列的数据类型计算兼容性
  3. 类型标准化：移除精度/长度（如 numeric(10,2) → numeric）
  4. 兼容性判断：根据类型族判断兼容程度
  5. 平均值：avg_compatibility = sum(compatibilities) / len(columns)
类型兼容性规则：
类型组合 兼容性得分 说明
完全相同 1.0 类型完全一致
整数族内部 1.0 int, bigint, smallint, serial 等
数值族内部 0.8 numeric, decimal, float, real 等
整数 ↔ 数值 0.6 部分兼容（可能损失精度）
字符串族内部 0.5 varchar, char, text 等（长度可能不同）
其他组合 0.0 不兼容

10.5 核心设计原则

🎯 设计原则：
  • LLM 语义理解：利用 LLM 理解表结构和字段语义，发现隐式关系
  • 数据驱动验证：通过数据库采样验证 LLM 推断的准确性
  • 外键直通：保留物理外键关系，与 LLM 推断关系合并
  • Domain 优化：支持按业务域过滤，减少不必要的 LLM 调用
  • 异步批处理：支持异步并发调用，提高大规模数据处理效率
  • 容错机制：LLM 调用失败自动重试，确保稳定性

10.6 使用建议

💡 最佳实践：
  • 混合使用：先运行 rel 获取基于规则的关系，再运行 rel_llm 补充遗漏的关系
  • Domain 过滤：对于大型数据库，使用 --domain 参数减少 LLM 调用次数
  • 异步模式：对于 50+ 张表，启用 use_async=true 提高效率
  • 阈值调整：根据实际情况调整 accept_threshold（0.60-0.70 之间）
  • 成本控制：使用 domain 过滤和合理的 batch_size 控制 API 成本

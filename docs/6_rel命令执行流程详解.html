1. 命令概述

metaweave metadata --step rel 命令用于自动发现数据库表之间的关系，包括外键直通关系和基于规则推断的隐式关系。

核心功能

  • ✅ 从 JSON 元数据文件中提取外键直通关系
  • ✅ 基于物理约束（PK/UK/Index）和逻辑主键生成候选关系
  • ✅ 通过数据库采样计算 4 维度评分（包含率、名称相似度、类型兼容性、Jaccard 指数）
  • ✅ 应用决策规则过滤低置信度关系并抑制冗余关系
  • ✅ 输出 JSON 和 Markdown 格式的关系发现报告

5.2.1 复合键匹配策略：两阶段匹配

CandidateGenerator._find_target_columns()
📁 metaweave/core/relationships/candidate_generator.py:224
在目标表中查找匹配的列组合，使用两阶段策略。
Stage 1: 特权模式（Privilege Mode）
  1. 适用条件：源表是 PK/UK/逻辑键
  2. 目标约束：检查目标表的 PK/UK/Index/逻辑键
  3. 匹配算法：穷举排列 + 较低的名称相似度阈值
  4. 短路机制：匹配成功立即返回
Stage 2: 动态同名匹配（Dynamic Same-Name）
  1. 适用条件：Stage 1 未找到匹配
  2. 匹配规则：大小写不敏感的列名匹配
  3. 类型检查：type_compatibility >= 配置阈值
  4. 过滤规则：源表不过滤，目标表区分物理/逻辑约束

5.2.2 穷举排列算法

CandidateGenerator._match_columns_as_set()
📁 metaweave/core/relationships/candidate_generator.py:353
使用 O(n! × n) 的穷举排列算法找到最佳匹配。
算法步骤：
  1. 源列过滤：完全不过滤（尊重所有约束）
  2. 目标列过滤：
    • 物理约束（PK/UK/Index）：不过滤
    • 逻辑约束：按 composite.exclude_semantic_roles 配置过滤（默认排除 metric，实际配置还排除 description、attribute、complex）
  3. 穷举排列：尝试所有可能的列顺序组合
  4. 逐对检查：每对列必须满足：
    • name_similarity >= min_name_similarity
    • type_compatibility >= min_type_compatibility
  5. 综合评分：0.5×avg_name_sim + 0.5×avg_type_compat
  6. 选择最优：返回综合得分最高的排列
💡 复合键排除角色配置：
composite.exclude_semantic_roles 配置会同时影响：
  • 1. 逻辑主键生成阶段（LogicalKeyDetector）：在生成复合逻辑主键时排除这些角色
  • 2. 候选匹配阶段（CandidateGenerator）：在匹配目标表逻辑约束时过滤这些角色
  • 3. 当前配置：["metric", "description", "attribute", "complex"]
💡 复合键示例：
源表 order_item 有复合主键 (order_id, product_id)
目标表 inventory 有索引 (product_id, warehouse_id)
→ Stage 1 会尝试匹配 product_id，但因列数不等而失败
→ Stage 2 会尝试动态同名匹配，但因 order_id 在目标表不存在而失败

5.3 生成单列候选

CandidateGenerator._generate_single_column_candidates()
📁 metaweave/core/relationships/candidate_generator.py:813
生成单列候选关系。
单列候选条件：
  1. 源列要求：必须有定义约束（PK/UK）或是逻辑主键
  2. 目标列要求：必须有约束（PK/UK/Index）或是逻辑主键

  4. 名称相似度：动态阈值（重要目标 >= 0.7，普通目标 >= 0.85）

5.3.1 单列候选生成流程

详细步骤：
  1. 遍历源表：检查每个表的所有列
  2. 源列筛选：
    • 检查是否有定义约束（PK/UK）
    • 检查是否为逻辑主键（confidence >= 0.8）
    • 必须至少满足一个条件
    • 不过滤语义角色（尊重物理约束和逻辑主键）
  3. 遍历目标表：对每个源列，遍历所有目标表的列
  4. 目标列筛选：
    • 物理约束（PK/UK/Index）：不过滤
    • 逻辑约束：按 exclude_semantic_roles 配置过滤
  5. 类型兼容性检查：type_compatibility >= 0.5
  6. 名称相似度检查：
    • 目标列是"重要列"（有约束/逻辑键）：threshold = 0.7
    • 目标列是"普通列"：threshold = 0.85
  7. FK 去重：检查是否已存在外键关系
  8. 确定候选类型：
    • single_defined_constraint_and_logical_pk
    • single_defined_constraint
    • single_logical_key

5.3.2 单列候选示例

💡 示例 1：主键到主键
源表 department 有主键 dept_id
目标表 employee 有列 dept_id（有索引）
→ 类型兼容：INTEGER ↔ INTEGER = 1.0
→ 名称相似度：dept_id ↔ dept_id = 1.0
→ 候选类型：single_defined_constraint
→ 生成候选
💡 示例 2：逻辑主键到索引列

目标表 order_item 有列 product_code（有索引）
→ 类型兼容：VARCHAR ↔ VARCHAR = 0.85

6.1 评分器初始化

6.2 评分维度（4维度评分体系）

维度 权重 说明
inclusion_rate 55% 源列值在目标列中的包含率（数据库采样）
name_similarity 20% 列名相似度（Levenshtein 或 Embedding）
type_compatibility 15% 类型兼容性（体现 JOIN 性能）
jaccard_index 10% Jaccard 相似度（辅助判断）
💡 综合评分公式： composite_score = 0.55×inclusion_rate + 0.20×name_similarity + 0.15×type_compatibility + 0.10×jaccard_index
⚠️ 已删除的维度： uniqueness（逻辑错误）、semantic_role_bonus（权重过小）。当前版本使用 4 维度评分体系。

6.3 数据库采样与评分

RelationshipScorer._sample_and_calculate_inclusion()
📁 metaweave/core/relationships/scorer.py:246
从数据库采样并计算包含率、Jaccard 指数、唯一性和 JOIN 倍率。
采样与计算流程：
  1. 从源表和目标表各采样 1000 行（默认值，来自配置 sampling.sample_size）
  2. 提取值集合（排除 NULL 值）
  3. 计算交集和并集
  4. 计算 inclusion_rate = |交集| / |源集合|
  5. 计算 jaccard_index = |交集| / |并集|
  6. 计算源列和目标列的唯一性
  7. 执行 JOIN COUNT 查询获取 JOIN 倍率
💡 采样配置： 评分阶段的采样行数统一使用 sampling.sample_size 配置（默认1000）， 而不是 relationships.sample_size（已废弃）。

  2. 低于阈值的候选直接丢弃
  3. 高于阈值的候选进入抑制规则检查

• 配置文件（decision.accept_threshold）：默认 0.65
• 代码硬编码默认值（未配置时）：0.80
• 建议：在配置文件中明确设置此值，避免混淆

7.3 抑制规则

DecisionEngine._apply_suppression()
📁 metaweave/core/relationships/decision_engine.py:102
应用抑制规则，避免复合关系和单列关系的冗余。
抑制逻辑：
  1. 按表对（source_table -> target_table）分组
  2. 如果存在复合关系，检查单列关系：
    • ✅ 单列有独立约束（PK/UK）→ 保留
    • ❌ 单列无独立约束 → 抑制
    • ⚠️ 注意：索引(Index)不属于独立约束
  3. 如果不存在复合关系，保留所有单列关系
⚠️ 抑制原则： 复合关系优先，单列关系只有在有独立约束时才能与复合关系共存。

9.4 核心设计原则

🎯 设计原则：
  • 尊重物理约束：源列的物理约束（PK/UK）完全不过滤
  • 数据驱动：通过数据库采样验证关系的真实性
  • 4维度评分：综合考虑包含率、名称、类型、Jaccard 四个维度
  • 避免冗余：复合关系优先，抑制无独立约束的单列关系
  • 确定性输出：relationship_id 基于 MD5 哈希，保证可重现
  • 灵活配置：支持 Levenshtein 和 Embedding 两种名称相似度算法
⚠️ 重要配置说明：
  • accept_threshold：
    • 配置文件默认值：0.65（decision.accept_threshold）
    • 代码硬编码默认值（未配置时）：0.80
    • 建议：在配置文件中明确设置此值，避免混淆
  • name_similarity：
    • string 方法（Levenshtein）：重要目标列建议 0.6，普通目标列建议 0.9
    • embedding 方法（语义向量）：重要目标列建议 0.9，普通目标列建议 0.9
  • sampling.sample_size：统一配置评分阶段的采样行数，替代已废弃的 relationships.sample_size

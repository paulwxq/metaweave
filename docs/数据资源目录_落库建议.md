# MetaWeave 数据资源目录（Web UI）落库建议

## 1. 背景与目标

你希望 MetaWeave 在“表分类 + 关系发现”的主目标不变的前提下，更充分复用与扩展中间结果（画像、统计、质量评估等），并基于这些信息做一个 Web UI（数据资源目录），支持：

- 按 domain/表类型/规模/质量等维度浏览与筛选
- 查看表/字段画像（统计信息、可解释指标）
- 展示与查询表间关系（含证据与置信度）
- 后续扩展：字典查询、质量规则、对比与版本等

## 2. 是否需要把文件中间结果写入数据库？

建议：**需要落库，但不建议替代文件输出**；采用“双轨制”最稳：

- `output/`：保留文件产物（导出/交换/回放/排障/离线）
- PostgreSQL：作为 Web UI / API 的主读存储（查询、筛选、版本、权限、对比、检索更适合）

这样既兼容当前流程（导出 CQL/关系等），又能把“画像/质量/字典”等做成可持续演进的数据产品。

## 3. 存储策略（PostgreSQL + JSONB）

总体原则：

- **稳定字段结构化**：可排序、可索引、可筛选的字段（domain、表类型、行数、更新时间、置信度等）
- **不稳定字段 JSONB**：后续可能变化/扩展的指标与证据（画像细项、规则命中详情、LLM 证据等）
- **版本化（run）**：一次执行对应一个 `run_id`，让 UI 支持“最新/指定版本/对比两次运行”

## 4. 表结构规划建议（核心）

### 4.1 两个基础概念：`source` 与 `run`

- `sources`：数据源标识（逻辑 ID），例如连接信息的抽象（host/db/schema 等）
- `runs`：一次 MetaWeave 执行（run_id、开始/结束时间、参数、config_hash、代码版本、是否 LLM 流程等）

价值：同一张表/字段可多次生成画像与关系，Web UI 可以做时间轴、对比与回溯。

### 4.2 资源目录主体（结构化优先）

- `tables`：表级元数据
  - schema/name/comment/domain/table_category（维/事实/桥接…）、row_estimate、size_bytes、last_analyzed 等
- `columns`：字段级元数据
  - table_id、name、data_type、nullable、comment、是否 identifier/description/enum 等

### 4.3 画像与统计（结构化 + JSONB）

- `table_profiles`：`run_id + table_id`
  - 结构化：row_count、column_count、sample_strategy、profile_time_ms…
  - JSONB：`profile`（扩展项：分布摘要、采样信息、质量摘要等）
- `column_profiles`：`run_id + column_id`
  - 结构化：null_rate、distinct_count、min/max、avg_len、topk_count…
  - JSONB：`profile`（扩展项：top_values、分桶、异常值、枚举候选证据等）

### 4.4 关系表（可解释性优先）

- `relationships`：`run_id + 边的唯一键`
  - 结构化：src/dst（表/字段）、relation_type（FK/推断/LLM）、confidence、domain_scope（同域/跨域）
  - JSONB：`evidence`（推断依据、打分细项、LLM 原因摘要、规则命中等）

### 4.5 文档/产物（artifact 索引）

- `artifacts`：记录每次 run 产物的“索引信息”
  - kind（json/json_llm/rel/cql/md/ddl…）、path、content_hash、size、table_id（可选）
  - 建议默认不把大文件全文写 DB，先存 `path + hash`；必要时再做“关键片段/摘要”入库。

### 4.6 质量评估（可选但推荐预留）

- `quality_metrics` / `quality_issues`
  - 表/字段维度的规则检测结果（空值率异常、疑似主键重复、枚举异常、长度异常等）
  - JSONB 保存规则详情，结构化保存 rule_id、severity、score 等

## 5. 执行步骤与“分阶段填表”映射（建议）

建议按 MetaWeave 的 step 逐步填充 DB（每个 step 都能幂等 upsert）：

- `ddl`：可写 `artifacts(kind=ddl)`；可选写入 `tables/columns` 的类型信息
- `json`：写 `tables/columns`（基础元数据）+ `table_profiles/column_profiles`（画像）
- `json_llm`：在 `tables/columns` 或 profile 中补充“LLM 注释/摘要”等；产物入 `artifacts`
- `rel`：写 `relationships`（非 LLM 推断）+ 产物入 `artifacts`
- `rel_llm`：写/更新 `relationships`（LLM 关系 + evidence）+ 产物入 `artifacts`
- `cql` / `cql_llm`：产物入 `artifacts`；（可选）记录“导出到 Neo4j 的版本”与摘要统计
- `md`：产物入 `artifacts`；（可选）写入 `table_docs`（或 artifacts+摘要）
- `load ...`：记录“加载任务”与结果（可选：`load_jobs`）

## 6. 推荐实施路径（最小可行 → 可持续演进）

### Phase 1（最小可用：支撑 Web 目录）

- 建表：`sources`、`runs`、`tables`、`columns`、`table_profiles`、`column_profiles`、`relationships`
- 支持从 `output/json` 与 `output/rel` 导入 DB（或生成时直接写 DB）
- Web UI 只读 DB：列表/筛选/详情/关系展示

### Phase 2（增强：可追溯与可解释）

- `artifacts`：记录产物路径与 hash（可回放、可对比）
- `quality_issues`：落地一批“轻量质量规则”
- 增加全文检索：表名/字段名/注释/域/标签（Postgres FTS 或 trigram）

### Phase 3（产品化：对比、权限、协作）

- 运行对比：两次 run 的画像/关系差分
- 权限/分组/标签体系
- 质量评分体系与可配置规则

## 7. 注意事项

- **幂等与增量**：每个 step 都应可重复执行；用 `run_id + 唯一键` upsert
- **体量与隐私**：尽量只存聚合指标；top_values 建议截断/脱敏；避免存真实样本
- **与 Neo4j 的关系**：Neo4j 仍可作为图执行/可视化后端；Postgres 作为目录与指标主存储，两者互补

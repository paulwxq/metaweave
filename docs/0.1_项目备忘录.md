# 项目备忘录

## 1. rel 步骤应支持 --domain 参数

**日期**: 2026-03-16

**背景**:
当前 `--domain` / `--cross-domain` 参数仅对 `rel_llm` 步骤生效，对 `rel` 步骤无效。`rel` 步骤使用 `RelationshipDiscoveryPipeline`，其内部的 `CandidateGenerator` 完全不接收 domain 参数。

**现状分析**:

| 步骤 | `--domain` 生效？ | 表对生成方式 |
|------|:---:|------|
| rel_llm | 有效 | `DomainResolver.resolve_table_pairs()` 按 domain 筛选 |
| rel | **无效** | `CandidateGenerator` 对所有表做 O(n²) 双重循环 |

`rel` 的 `CandidateGenerator._generate_composite_candidates()` 和 `_generate_single_column_candidates()` 内部都是 `for source in tables → for target in tables` 的全量遍历（见 `candidate_generator.py` 行 111-125），与 `rel_llm` 面临相同的组合爆炸问题。

**建议在 rel 中启用 --domain 的原因**:

1. **一致性**: `rel` 和 `rel_llm` 做的是同一件事（发现表间关系），只是方法不同（规则 vs LLM）。用户合理期望 `--domain` 对两者行为一致。

2. **性能收益**: `rel` 步骤虽然单次配对成本低于 `rel_llm`（不调用 LLM），但每对仍需数据库采样和评分。15 张表全量是 C(15,2)=105 对，按 domain 筛选后可能只需 20-30 对，减少了大量不必要的数据库查询。

3. **结果质量**: 不相关的表之间（如「地理信息」×「财务结算」）即使通过规则匹配到同名列（如 `id`、`last_update`），这些候选关系也多半会被评分淘汰，属于无效计算。domain 过滤能从源头减少这类噪音。

4. **实现路径清晰**: 已有 `DomainResolver` 组件，只需在 `RelationshipDiscoveryPipeline.__init__` 中接收 `domain_resolver` 参数，在 `CandidateGenerator` 中增加表对过滤逻辑即可。改动局部且可控。

**实现要点**:

- `RelationshipDiscoveryPipeline.__init__` 新增 `domain_resolver`、`domain_filter`、`cross_domain` 参数
- `CandidateGenerator` 接收一个可选的 `allowed_pairs: Set[Tuple[str, str]]` 参数，在双重循环中跳过不在集合内的表对
- CLI 层 `metadata_cli.py` 的 `rel` 步骤和 `all` 流程中的 `rel` 子步骤传递 domain 相关参数
- 不传 `--domain` 时行为不变（全量遍历）
